## 项目概述

ragtag_crew 是单用户自托管 AI 编程助手，当前提供 Telegram、微信两个正式入口，并保留 REPL 作为开发 / 调试入口。

- 语言：Python 3.12（上限 `<3.13`），包管理用 uv
- 框架：python-telegram-bot + litellm，无第三方 agent SDK
- 入口：`src/ragtag_crew/`，测试：`tests/`
- 运行：`uv run ragtag-crew` 或 `uv run python -m ragtag_crew.main`

## 代码风格

- 中文注释和 commit message
- 不加无意义注释；注释说明「为什么」而非「做什么」
- 类型注解覆盖函数签名，尽量用 builtin 类型（`str | None` 而非 `Optional[str]`）
- 异步优先：IO 操作用 `async/await`，避免阻塞事件循环
- 配置统一走 `pydantic-settings`，不硬编码

## 实施原则

1. **先确认，再编码**
- 不替用户脑补未明确的需求或边界条件
- 有歧义时先说明理解或提出简短澄清，不静默选择一种解释
- 如果存在明显更简单的实现路径，应先指出，不默认走复杂方案

2. **最小正确改动**
- 默认选择能满足需求的最小改动，不提前做多前端、多层抽象或投机性扩展
- 不为单次使用引入额外抽象、配置项、兼容层或框架化封装
- 没有明确收益时，不做“顺手优化”

3. **只改和需求直接相关的代码**
- 不改无关注释、格式、邻近逻辑或历史遗留问题
- 只清理本次改动引入的冗余；发现既有问题可说明，但不要未经要求顺手修复
- 每个改动文件都应能直接映射到本次需求

4. **用可验证结果驱动实现**
- 非平凡任务先明确验收方式，再动手实现
- bug 修复优先补复现测试；功能改动优先补行为测试
- 完成后至少运行相关测试；影响面较大时运行 `uv run pytest tests/`

## 审查重点

请侧重以下方面，按严重程度降序：

1. **错误**：逻辑缺陷、异常处理遗漏、类型不匹配、竞态条件
2. **安全**：路径穿越、命令注入、未校验的外部输入、敏感信息泄露
3. **健壮性**：边界条件、超时处理、资源释放、并发安全
4. **可维护性**：过度抽象、重复代码、命名不清

## 不需要审查的

- 纯排版调整（空行、import 排序）
- 中文措辞优化
- docstring 格式
