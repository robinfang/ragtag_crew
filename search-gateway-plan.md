# `ragtag_crew` 搜索与 Gateway 专项设计

## 1. 目标

搜索能力不应只是一把临时工具。

`ragtag_crew` 接下来需要同时满足两件事：

- 自己带一个默认可用、单机可部署、可离线降级的搜索能力
- 为未来独立的 search gateway / portal 预留稳定接入面

这意味着搜索要拆成两层，而不是全部堆进当前 agent 进程里。

## 2. 两层架构

### 2.1 项目内置搜索层

这一层直接跟 `ragtag_crew` 一起交付，负责兜底可用性。

建议范围：

- 本地代码与文件搜索：`grep` / `find` / `ls`
- Windows 下的 `Everything` 搜索适配
- 后续可补一个简单联网搜索 API

这一层的要求是：

- 默认可用
- 配置简单
- 能单机部署
- 不依赖额外基础设施

### 2.2 外部搜索 Gateway 层

这一层单独演进，不与当前 bot 进程耦合。

建议职责：

- 多 provider 池管理
- 查询路由
- 失败切换
- 去重与融合
- 重排与来源标注
- 缓存、配额、限流、审计

这一层的要求不是“先做出来”，而是“先把接口边界想清楚”。

## 3. 为什么现在不直接把所有搜索都塞进本项目

- 多源搜索 provider 的可用性差异很大
- 认证、代理、配额和故障处理会快速膨胀
- 聚合与去重逻辑不适合和 Telegram handler、tool registry 混在一起
- 将来如果还要把搜索能力暴露给别的 agent、CLI 或 Web，单独 gateway 更稳

所以当前项目内只保留“基础版 / 兜底版搜索”，复杂多源能力尽量外置。

## 4. 推荐接口形态

推荐顺序不是“先 MCP 后 OpenAPI”，而是：

`search core -> OpenAPI -> MCP adapter`

原因：

- `OpenAPI` 更容易调试、版本化、观测和做错误归因
- `MCP` 更适合做 agent 工具暴露层，而不是最底层业务协议
- 未来 `ragtag_crew` 本体也能优先消费固定 `OpenAPI`，再按需要接 MCP 适配器

## 5. 对 `ragtag_crew` 的直接要求

当前项目侧应该做到：

1. 工具系统允许多个来源并存
2. 搜索工具只是工具来源的一种，而不是特殊硬编码分支
3. Windows `Everything` 被当作平台适配器，不混入通用 shell
4. 将来接 gateway 时，最好只需要新增一个 provider，而不是重写 agent loop

## 6. 当前阶段建议

当前阶段先做三件事：

1. 建立阶段 1 的外部能力接入层
2. 接入 Windows `Everything` 作为平台搜索适配器
3. 接入 `MCP client`，为后续外部工具打通路径

联网多源搜索 gateway 先写清楚边界，不在这一轮直接做完。

## 7. 第一版边界

第一版不做：

- 多 provider 搜索聚合
- 搜索结果 rerank
- 搜索缓存层
- 搜索审计后台
- 动态 `OpenAPI spec` 导入
- 自动把搜索结果写入长期记忆

第一版只做：

- 本项目内的基础搜索能力继续可用
- `Everything` 能作为 Windows 搜索工具挂到工具系统里
- `MCP client` 能把外部工具纳入统一工具系统

## 8. 与上下文系统的关系

搜索系统和上下文系统要分层：

- 搜索负责“找到东西”
- 上下文系统负责“哪些结果进入当前轮、哪些结果进入摘要、哪些结果应该进记忆”

这意味着现在先把外部能力接入层做稳，之后再让 `session_summary`、`memory`、`context builder` 去承接这些结果。

## 9. 当前结论

`ragtag_crew` 需要：

- 内置一个基础可用搜索层
- 规划一个独立 search gateway / portal
- 用统一工具系统同时承接本地工具、平台适配器、MCP 工具、未来固定 OpenAPI 工具

这就是当前阶段 1 的架构落点。
