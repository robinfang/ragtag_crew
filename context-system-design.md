# `ragtag_crew` 上下文系统设计草案

## 1. 文档目的

本文档用于定义 `ragtag_crew` 的上下文系统，目标不是一次性把所有“记忆 / prompt / skill / 会话管理”功能都做完，而是先回答下面几个基础问题：

1. 哪些信息应该进入模型上下文。
2. 哪些信息应该长期保存，但不必每轮都注入。
3. 哪些信息属于临时会话状态，哪些属于跨会话记忆。
4. 未来如果做 `/memory`、`/prompt`、图片输入、多模型验证，应该挂在哪一层，而不是继续把逻辑堆在 `AgentSession` 里。

本文档会同时给出：

- 建议采用的分层模型
- 当前实现与目标状态的差距
- 需要后续讨论和拍板的决策点

## 2. 设计背景

### 2.1 当前项目状态

当前 `ragtag_crew` 已经具备以下能力：

- Python 自建 agent loop
- Telegram 单入口
- `read` / `write` / `edit` / `bash` / `grep` / `find` / `ls` 七个工具
- HTML 富文本输出
- 会话 JSON 持久化
- 本地 Markdown skill 的会话级启用

当前上下文的实际构造方式比较简单：

- `src/ragtag_crew/agent.py`
  - 当前会把 `system_prompt`
  - 加上已启用的 skill 正文
  - 再加上 `session.messages`
  - 一起传给 `stream_chat()`
- `src/ragtag_crew/telegram/session_store.py`
  - 当前会持久化 `model`、`tool_preset`、`enabled_skills`、`system_prompt`、`messages`

这说明基础设施已经有了，但上下文仍然只有两层：

- 一层是“系统提示 + skill”
- 一层是“完整会话消息”

这对于早期阶段是够用的，但它还不足以支撑：

- 长期记忆
- 自动压缩后的恢复
- 会话级临时规则与长期偏好的区分
- 后续多入口扩展

### 2.2 为什么现在需要上下文系统设计

目前的主要风险不是“功能太少”，而是“不同类型的信息还没有分层”。

典型例子：

- 用户长期偏好
- 项目约定
- 当前会话临时要求
- skill 模板
- 上一轮工具输出

这些内容最终都可能影响模型回答，但它们的生命周期、来源、体量、是否应长期保存都不同。

如果继续把这些内容直接塞进一个 `system_prompt` 字符串或 `messages` 列表，后面会出现三个问题：

1. 上下文越来越大，质量越来越差。
2. 很难回答“模型为什么这次这样回复”。
3. `memory`、`prompt`、`skill` 的职责会越来越混。

## 3. 外部参考结论

### 3.1 Claude Code 官方文档

从官方文档里可以提炼出四个关键原则：

1. **长上下文会退化**。
   - 上下文窗口不是越大越好。
   - 旧内容如果不整理，会出现上下文质量下降。

2. **长期指令和自动记忆是两套来源，但都只是上下文，不是硬约束**。
   - `CLAUDE.md`：人写的长期指令
   - auto memory：系统自动积累的长期记忆

3. **长期记忆不应全部无脑注入**。
   - `MEMORY.md` 只加载前一部分
   - 更细的主题文件按需读取

4. **压缩不是记忆替代品**。
   - compaction 负责保留会话连续性
   - 结构化记忆负责跨会话保留重要信息

### 3.2 OpenClaw 官方文档

OpenClaw 的做法更体系化，最值得借鉴的是它的分工，不是它的全部形态。

它把内容分成：

- OpenClaw 自己构建的 system prompt
- 注入的工作区文件
- 会话历史
- 工具调用和工具结果
- 长期记忆 `MEMORY.md`
- 日记式记忆 `memory/YYYY-MM-DD.md`
- 技能列表只注入元数据，技能正文按需读取
- 旧工具结果可以修剪，但不改写原始会话记录
- 压缩前会尝试做一次记忆落盘

对本项目最有价值的原则是：

1. **长期记忆和当前会话必须分开。**
2. **工具结果不是记忆。**
3. **不是所有文件都该每轮注入。**
4. **按需读取比全量注入更重要。**

### 3.3 `claw-code` 项目

本次只参考公开仓库可见的结构与实现思路，不把它当成唯一标准。

本次调研中确认到的线索：

- 仓库根有 `CLAW.md`
- 运行时 Rust 目录中有：
  - `prompt.rs`
  - `session.rs`
  - `compact.rs`
- Python 树中有：
  - `context.py`
  - `session_store.py`
  - `transcript.py`
  - `memdir/`

其中比较值得借鉴的点：

1. **指令文件发现是独立模块。**
   - `prompt.rs` 单独负责发现 `CLAW.md` / `CLAW.local.md` / `.claw/...`
   - 说明“提示词组装”和“业务逻辑”应当分开

2. **系统提示词有稳定边界。**
   - `SYSTEM_PROMPT_DYNAMIC_BOUNDARY`
   - 说明 prompt 内部也应区分“稳定部分”和“动态部分”

3. **会话结构是显式数据模型。**
   - `session.rs` 明确区分 `role`、`blocks`、`usage`
   - 说明会话持久化不应只是“随便 dump 一个列表”

4. **压缩是单独模块。**
   - `compact.rs` 独立处理“保留最近消息 + 总结更早消息”
   - 说明 compaction 应当作为会话层能力，而不是揉进 Telegram handler

`claw-code` 当前公开内容并没有给出完整的“长期记忆分层”设计说明，但它对我们有一个很重要的提醒：

> prompt、session、compact、context 这四件事应该是独立职责，而不是一个大函数顺手拼起来。

### 3.4 `opencode-dynamic-context-pruning` 插件

这个项目与前面几个参考的定位不同。

它不是长期记忆系统，也不是项目背景管理系统，而是一个面向 OpenCode 的 **动态上下文裁剪插件**。

它重点解决的问题是：

- 长会话里哪些旧消息已经可以压缩
- 哪些旧工具输出可以替换成占位内容
- 哪些报错工具输入在若干轮之后可以清理
- 如何在不改写原始会话的前提下，把“裁剪状态”叠加到实际发送给模型的上下文上

本次调研中提炼出的几个关键点如下。

#### 3.4.1 它做得好的地方

1. **把“压缩 / 裁剪”建成独立能力，而不是零散补丁。**
   - 插件入口把 system prompt 注入、messages transform、command、event、tool 注册分开处理。
   - 说明“上下文裁剪”值得单独建模，而不是塞进会话主循环里顺手处理。

2. **压缩状态是显式结构，不是只有一段摘要字符串。**
   - 它为每个 compression block 记录 `blockId`、`runId`、`startId`、`endId`、`anchorMessageId`、`effectiveMessageIds`、`effectiveToolIds`、`summaryTokens`、`compressedTokens`、`durationMs` 等字段。
   - 说明会话压缩如果要长期可维护，最好有独立的 block/state 模型，而不是只留一个 `session_summary` 文本。

3. **保护机制做得很明确。**
   - protected tools
   - protected file patterns
   - 可选保护 user messages
   - 说明裁剪系统必须先定义“什么绝不能丢”，再讨论“什么可以删”。

4. **自动裁剪不是立刻强删，而是“阈值 + nudge + 手动命令”混合。**
   - 它根据 token 使用量和阈值判断是否该提醒模型压缩。
   - 既支持自动注入压缩提醒，也支持 `/dcp compress` 这类手动触发。
   - 说明压缩不一定要第一版就做全自动，可以先做“状态 + 提醒 + 手动整理”。

5. **会话原文与裁剪状态分开存。**
   - 原始消息不直接改写。
   - prune/compress 状态另行持久化。
   - 这对本项目很有启发：长期记忆、会话原文、压缩状态最好是三种不同数据。

#### 3.4.2 它不适合直接照搬的地方

1. **它强依赖 OpenCode 的 plugin hook。**
   - `experimental.chat.system.transform`
   - `experimental.chat.messages.transform`
   - `command.execute.before`
   - `event`
   - 这些机制不适用于当前的 Telegram Python 架构。

2. **它解决的是“会话裁剪”，不是“长期记忆分层”。**
   - 它并不负责 `PROJECT.md`、`USER.local.md`、`MEMORY.md` 这类文件的注入与边界。
   - 所以不能把它当作记忆层设计的替代品。

3. **它的许多机制建立在 OpenCode 会话元数据之上。**
   - message ID 注入
   - tool call ID
   - assistant token usage 回传
   - 这些在当前项目中并不存在同等接口。

4. **许可证与集成方式也不适合直接复用代码。**
   - 该项目采用 `AGPL-3.0-or-later`。
   - 当前更适合借鉴思路，而不是直接搬实现。

#### 3.4.3 对 `ragtag_crew` 的直接启发

从本项目角度，最值得采纳的不是它的插件外形，而是下面四个原则：

1. **context builder / session store / memory store / compression state 应分开。**
2. **未来做会话压缩时，应引入显式 block 状态，而不是只靠一段总摘要。**
3. **压缩前必须先定义 protected content 规则。**
4. **自动压缩应建立在阈值、提醒和人工介入之上，而不是突然硬截断。**

因此，本项目应把 `opencode-dynamic-context-pruning` 视为：

> **会话压缩层的参考实现**，而不是长期记忆层的参考实现。

### 3.5 `qmd` 项目

`qmd` 的定位与前面几个参考也不同。

它本质上不是会话上下文管理器，也不是长期记忆系统，而是一个 **本地知识库检索引擎**。

它主要提供：

- BM25 全文检索
- 向量语义检索
- LLM rerank
- 基于 MCP / CLI / SDK 的文档读取接口
- 路径级 `context` 标签继承

这类系统解决的问题是：

- 当知识库规模变大后，如何先找到相关文档
- 如何只把命中的正文片段带回当前会话
- 如何为命中文档附带更稳定的“语义标签”或“目录说明”

#### 3.5.1 它做得好的地方

1. **把“检索”和“上下文注入”分开。**
   - 先 `search/query`
   - 再 `get/multi_get`
   - 说明大型知识库不应默认整包注入 prompt，而应先检索再取正文。

2. **路径级 context 设计很实用。**
   - 支持 global context
   - 支持 collection 根 context
   - 支持子路径 context
   - 命中文件时按路径前缀继承并拼接
   - 这对本项目很有价值，因为 `memory/` 或未来的文档库也可能需要类似的“目录语义标签”。

3. **返回结果时保留 `context` 字段。**
   - 说明检索系统不只是返回文件路径，还会返回“这份材料属于什么语境”。
   - 对 agent 来说，这能显著降低“拿到一段正文却不知道它是什么”的风险。

4. **接入形态清晰。**
   - CLI 可直接给人和 agent 用
   - MCP 适合更紧的工具集成
   - SDK 适合应用内嵌
   - 说明未来如果本项目引入外部检索层，不一定非要自己从零写。

#### 3.5.2 它不适合直接替代的部分

1. **它不负责会话状态。**
   - 不处理 `session_summary`
   - 不处理压缩状态
   - 不处理当前轮 messages 的裁剪与装配

2. **它不负责长期记忆分层本身。**
   - 它可以检索 `MEMORY.md` 或 `memory/*.md`
   - 但不会替我们定义 `PROJECT.md / USER.local.md / MEMORY.md / session` 的边界

3. **它更适合“知识库规模已变大”的阶段。**
   - 当前项目更紧迫的是把上下文装配、记忆分层、会话摘要跑通
   - 不是立刻引入一个完整的本地 hybrid retrieval 栈

4. **运行成本更高。**
   - 本地 SQLite + embedding + rerank + GGUF 模型
   - 适合作为独立 retrieval 子系统
   - 不适合作为当前最小上下文系统的前置依赖

#### 3.5.3 对 `ragtag_crew` 的直接启发

从本项目角度，更合理的定位是：

> `qmd` 适合作为未来的 **外部检索层 / 知识库检索后端**，而不是当前上下文主装配逻辑的一部分。

这意味着：

1. `PROJECT.md` / `USER.local.md` / `MEMORY.md` 仍应由本项目自己的 context builder 直接装配。
2. `memory/*.md` 正文、归档会话、项目文档、会议记录等材料，未来可以交给 `qmd` 这类系统按需检索。
3. 如果将来接入，推荐让它服务于：
   - `/memory search`
   - `/kb query`
   - agent 内部的外部资料检索
4. 不建议让它接管当前会话层的 system prompt 组装。

因此，本项目应把 `qmd` 视为：

> **可选的外部 retrieval 层参考**，不是长期记忆层或会话压缩层的替代品。

## 4. 设计目标

本项目的上下文系统应满足以下目标：

1. **单用户、单 Telegram 入口优先**。
   - 不为了未来的多端支持提前做过重抽象。

2. **把不同生命周期的信息分开管理**。
   - 长期规则
   - 长期记忆
   - 场景模板
   - 当前会话
   - 临时证据

3. **默认上下文尽量短**。
   - 只有真正值得每轮注入的内容才常驻。

4. **详细内容按需读取**。
   - 包括记忆正文、长工具输出、历史细节。

5. **上下文来源可解释**。
   - 后续应该能回答“这条回答受哪些上下文影响”。

6. **为后续功能留接口**。
   - 记忆系统
   - 会话 prompt 自定义
   - 压缩
   - 图片/文件输入

## 5. 上下文分层模型

建议将 `ragtag_crew` 的上下文划分为 **6 层**。

### 第 1 层：基础系统层

这是代码内置的、最稳定的一层。

**作用：**

- 定义 agent 身份
- 定义工具能力
- 定义安全边界
- 定义 Telegram 场景下的默认行为

**示例内容：**

- “你是本地部署的 Telegram coding agent”
- “工具调用要谨慎，不要越过工作目录边界”
- “如未验证，不要声称已验证”

**来源：**

- 代码生成

**是否每轮注入：**

- 是

**设计要求：**

- 短
- 稳定
- 不混入用户偏好和项目细节

### 第 2 层：启动背景层

这是“会话一开始就应该知道”的静态背景。

**作用：**

- 提供项目级和用户级的稳定背景

**建议内容：**

- 项目定位
- 仓库约定
- 用户长期工作偏好

**推荐文件：**

- `PROJECT.md`：项目背景与约定
- `USER.md` 或 `USER.local.md`：用户偏好

**是否每轮注入：**

- 是，但文件必须短

**说明：**

这一层对应 Claude Code 的 `CLAUDE.md`，也部分对应 OpenClaw 的 `IDENTITY.md` / `USER.md` / `AGENTS.md` 的组合，但本项目不建议一开始拆成很多文件。

### 第 3 层：长期记忆层

这是跨会话持续存在的信息。

建议再细分成两块。

#### 3A. 长期记忆索引

**推荐文件：**

- `MEMORY.md`

**内容类型：**

- 项目长期决策
- 用户长期偏好
- 重要约定的短摘要
- 指向更详细记忆的索引

**是否每轮注入：**

- 是，但必须严格控长

**作用：**

- 提供长期背景摘要
- 告诉系统“详细信息在哪”

#### 3B. 长期记忆正文

**推荐目录：**

- `memory/`

**建议组织方式：**

- `memory/preferences.md`
- `memory/project-decisions.md`
- 或 `memory/YYYY-MM-DD.md` 这类日记文件

**是否每轮注入：**

- 否

**使用方式：**

- 检索后按需读取

**核心原则：**

> 长期记忆的正文不应默认全部进入上下文。

### 第 4 层：场景控制层

这是“当前应该怎样工作”的控制层。

它包含三类内容。

#### 4A. Skill

**来源：**

- `skills/*.md`

**作用：**

- 复用固定场景模板
- 如 review、debug、docs

**当前状态：**

- 已实现会话级启用

**Meta-Harness 启示：**

论文发现 skill 描述文本是搜索质量最强的杠杆。后续 skill 设计应注重：

- 描述文本清晰表达 skill 的适用场景和边界
- 包含足够的关键词，使任务路由能准确匹配
- 避免模糊或过于宽泛的描述

#### 4B. 会话级临时规则

这部分就是未来如果做 `/prompt`，它应该归属的层。

**作用：**

- 只影响当前会话
- 例如"这次回答简短一点""这轮只分析不改代码"

**当前状态：**

- 尚未实现

#### 4C. 环境引导信息（env bootstrap）

受 Meta-Harness 启发新增的子层。

**来源：**

- 每次会话启动时自动采集

**作用：**

- 在 agent 循环开始前，采集当前工作环境的快照并注入提示词
- 避免模型花费 2-4 轮探索性调用去理解环境

**建议采集内容：**

- 当前工作目录与文件树概览
- 可用语言与版本
- 包管理器状态
- 系统内存与磁盘状态

**实现要求：**

- 15 秒超时，静默失败
- 采集结果不应超过上下文预算的 5%
- 仅在首次请求或环境变更时重新采集

**重要结论：**

- `skill` 不是长期记忆
- `session prompt` 不是长期记忆
- 环境引导信息是会话级临时快照，不是持久配置
- 它们都属于"当前工作模式"，但生命周期比长期记忆短

### 第 5 层：会话工作层

这是当前 chat 的运行时上下文。

**内容：**

- 当前消息历史
- 当前模型
- 当前工具预设
- 当前启用 skill
- 未来可能包括会话级 prompt
- 压缩后的会话摘要

**当前状态：**

- 已有 JSON 持久化
- 当前直接持久化全部 `messages`

**特点：**

- 可重置
- 可过期
- 可压缩
- 不等于长期记忆

### 第 6 层：临时证据层

这是最容易把上下文撑爆的一层。

**内容：**

- 文件读取结果
- `grep/find/ls` 输出
- `bash` 输出
- 后续的图片/文件解析结果
- 当前轮推理直接依赖的材料

**原则：**

- 这是“证据”，不是“记忆”
- 默认应当短命
- 需要裁剪和淘汰
- 只有被确认有长期价值时，才提升到记忆层

## 6. 推荐的上下文组装顺序

建议最终给模型的上下文按下面顺序组装：

1. 基础系统层
2. 启动背景层
3. 长期记忆索引层（`MEMORY.md`）
4. 当前启用的 skill
5. 当前会话级临时规则（若未来实现 `/prompt`）
6. 会话摘要
7. 最近若干轮会话历史
8. 本轮按需取回的记忆正文
9. 本轮按需取回的工具结果 / 文件结果 / 附件结果

这样做的原因：

- 越稳定的内容越靠前
- 越临时的内容越靠后
- 长期信息和临时证据不会混在一起
- 出问题时比较容易追溯

## 7. 推荐的文件与数据结构

## 7.1 文件层

建议后续逐步引入以下文件结构：

```text
ragtag_crew/
├── PROJECT.md                 # 项目级长期背景
├── USER.local.md              # 用户级长期偏好，默认 gitignore
├── MEMORY.md                  # 长期记忆索引，短摘要
├── memory/
│   ├── preferences.md         # 用户偏好正文
│   ├── project-decisions.md   # 项目决策正文
│   └── YYYY-MM-DD.md          # 运行日志 / 日记式记忆
├── skills/
│   └── *.md                   # 场景模板
└── data/sessions/
    └── <chat_id>.json         # 会话状态
```

### 7.2 当前会话 JSON

建议未来逐步从“简单消息列表持久化”演进成更明确的数据结构，至少区分：

- `model`
- `tool_preset`
- `enabled_skills`
- `session_prompt`（如果未来实现）
- `session_summary`
- `messages`
- `last_active_at`

其中：

- `session_summary` 用于承接 compaction
- `messages` 只保留最近窗口内必要内容

## 8. 推荐的读取策略

### 8.1 默认每轮注入

默认每轮都带的内容建议控制在以下范围：

- 基础系统提示
- `PROJECT.md`
- `USER.local.md`（如果存在）
- `MEMORY.md`
- 当前启用的 skill
- 当前会话摘要 + 最近若干轮消息

### 8.2 按需读取

以下内容默认不要全量注入：

- `memory/*.md` 正文
- 长工具输出
- 历史会话老消息
- 图片 / 文件解析内容

### 8.3 压缩与修剪

建议未来引入两种不同机制：

1. **会话压缩**
   - 作用对象：旧会话消息
   - 输出：`session_summary`
   - 目的：保留连续性

2. **证据修剪**
   - 作用对象：旧工具结果
   - 目的：减少上下文膨胀
   - 不应改写原始持久化记录

## 9. 对 `memory`、`skill`、`prompt` 的边界定义

### 9.1 Skill

**定义：** 可复用的场景模板。

**适合存放：**

- review 时的关注点
- debug 时的工作方式
- docs 场景下的写法

**不适合存放：**

- 用户长期偏好
- 当前一次性临时要求

### 9.2 会话 prompt

**定义：** 当前会话的临时附加规则。

**适合存放：**

- “这次回答简短一点”
- “这轮只分析，不动代码”

**不适合存放：**

- 长期偏好
- 项目长期约定

### 9.3 记忆系统

**定义：** 跨会话长期保存的信息。

**适合存放：**

- 长期偏好
- 项目决策
- 已确认的稳定工作方式

**不适合存放：**

- 本轮临时指令
- 大段原始工具输出
- 可从代码即时重新推导出来的低价值事实

## 10. 当前实现与目标模型的差距

当前已经具备：

- 会话层：有
- skill 层：有
- 基础系统层：有

当前缺失：

- 启动背景层的外部文件化设计
- 长期记忆索引与正文分层
- 会话摘要字段
- 独立的上下文装配器
- 工具结果修剪
- 压缩前的记忆落盘策略
- 会话压缩状态的显式 block 化设计
- 裁剪时的 protected content 规则

## 11. 决策点清单

下面这些问题建议单独讨论，不要在实现时边写边猜。

### D1. 启动背景层是否拆成 `PROJECT.md` + `USER.local.md`

**建议：** 是。

**理由：**

- 项目约定和个人偏好是两种不同来源
- 单用户部署也需要分清“仓库内共享”和“机器本地私有”

### D2. `MEMORY.md` 是否每轮注入

**建议：** 是，但必须严格控长。

**理由：**

- 它应该只承担“记忆索引”角色
- 详细内容通过 `memory/*.md` 按需读取

### D3. `memory/YYYY-MM-DD.md` 是否默认自动注入“今天 + 昨天”

**建议：** 第一版不要。

**理由：**

- 本项目当前没有上下文大小监控和成熟修剪策略
- 直接自动注入很容易带来噪音
- 更稳妥的做法是检索后按需读

### D4. 是否立即实现 `/prompt`

**建议：** 暂缓。

**理由：**

- 在长期记忆层未落地前，`/prompt` 很容易和 `skill`、长期偏好混淆
- 它更适合放在上下文分层完成之后再实现

### D5. 记忆写入是手动还是自动

**建议：** 第一版以手动为主，自动为辅。

**建议形式：**

- 先做显式写入命令
- 之后再考虑在压缩前做“自动记忆刷新”

### D6. 会话压缩摘要是否放入 `session.json`

**建议：** 是。

**理由：**

- 这样 compaction 就是会话层能力
- 不需要在 Telegram 层临时拼补救逻辑

### D7. 是否需要引入独立的上下文装配器

**建议：** 是。

**建议模块：**

- `context_builder.py` 或 `context/` 子模块

**理由：**

- 让 `AgentSession` 只关心循环，不关心所有上下文来源的发现和拼接
- 对照 `claw-code` 的做法，这一职责应当独立

### D8. 会话压缩是否采用显式 block 状态，而不是只有 `session_summary`

**建议：** 是，但分阶段落地。

**第一阶段：**

- 先有 `session_summary`
- 先把“最近消息 + 旧消息摘要”跑通

**第二阶段：**

- 再引入 compression block / pruned range / protected content 这类显式结构

**理由：**

- `opencode-dynamic-context-pruning` 说明，只靠单一摘要字段很难支持可追踪、可恢复、可保护的裁剪逻辑
- 但当前项目还在 Telegram 单入口阶段，没必要一上来就复制完整复杂度

### D9. 是否在后期引入独立外部检索层（如 `qmd`）

**建议：** 暂不作为当前阶段前置依赖，但保留后续接入空间。

**理由：**

- 当前最重要的是把本项目自己的 context builder、memory 分层、session summary 先做稳。
- 当 `memory/`、项目文档、会话归档明显增多后，再引入检索层会更划算。
- `qmd` 这类系统更适合解决“如何从大量资料中找回相关正文”，而不是“如何定义本项目的上下文边界”。

## 11.1 会话压缩数据结构草案

如果后续进入“会话摘要与压缩”阶段，建议不要只增加一个 `session_summary: str` 字段，而是按下面两个层次建模。

### A. 第一阶段的最小结构

先满足“能压缩、能恢复运行、能持久化”的最低要求。

建议在当前 session JSON 中增加：

- `session_summary`
- `summary_updated_at`
- `recent_message_count`

推荐含义：

- `session_summary`
  - 保存“较早会话内容”的高保真摘要
  - 由系统在压缩时写入
- `summary_updated_at`
  - 标记最近一次压缩时间
  - 便于调试和后续决策
- `recent_message_count`
  - 记录当前保留了多少条最近消息
  - 让恢复逻辑更可解释

这阶段的 session JSON 形态可以是：

```json
{
  "model": "openai/GLM-5.1",
  "tool_preset": "coding",
  "enabled_skills": ["review"],
  "session_prompt": "",
  "session_summary": "...",
  "summary_updated_at": 1712345678.0,
  "recent_message_count": 12,
  "messages": [...],
  "last_active_at": 1712348901.0
}
```

### B. 第二阶段的 block 化结构

如果后面需要支持更可控的压缩、受保护内容和恢复能力，再从最小结构演进到显式 block 状态。

建议增加一个独立字段：

- `compression_state`

其内部建议至少包含：

- `blocks`
- `active_block_ids`
- `next_block_id`
- `protected_message_ids`
- `protected_tool_call_ids`
- `pruned_ranges`

推荐草案如下：

```json
{
  "compression_state": {
    "next_block_id": 3,
    "active_block_ids": [2],
    "blocks": [
      {
        "block_id": 1,
        "active": false,
        "topic": "早期需求讨论",
        "start_message_index": 0,
        "end_message_index": 18,
        "summary": "...",
        "created_at": 1712345000.0,
        "consumed_by_block_id": 2,
        "protected_items": {
          "message_ids": [1, 2],
          "tool_call_ids": []
        }
      },
      {
        "block_id": 2,
        "active": true,
        "topic": "上下文分层设计",
        "start_message_index": 0,
        "end_message_index": 31,
        "summary": "...",
        "created_at": 1712345600.0,
        "consumed_block_ids": [1],
        "protected_items": {
          "message_ids": [19],
          "tool_call_ids": ["memory_add_7"]
        }
      }
    ],
    "pruned_ranges": [
      {
        "start_message_index": 0,
        "end_message_index": 31,
        "active_block_id": 2
      }
    ]
  }
}
```

这不是要求第一版就全部实现，而是为了提前把“将来会往哪个方向长”写清楚。

## 11.2 protected content 规则草案

会话压缩不是“旧内容都可以删”。

建议先明确三类默认受保护内容：

### A. 长期有效的用户意图

例如：

- 当前阶段明确排除的功能范围
- 用户拍板的实现路线
- 当前任务的验收条件

这类内容如果被压缩，摘要里必须保留，不允许直接丢弃。

### B. 会影响后续操作安全性的内容

例如：

- 工作目录边界
- 模型 / API / endpoint 的关键配置
- 明确的权限限制

这类内容不一定要原文保留，但必须在摘要中稳定保留。

### C. 不可轻易重建的关键证据

例如：

- 已确认的外部调研结论
- 已跑过的关键验证结果
- 用户明确认可的架构决策

这类内容适合“摘要保留”，而不是“直接删除”。

相反，下面这些内容默认应该优先成为裁剪对象：

- 大段重复工具输出
- 已被更新结论覆盖的探索过程
- 报错后已无参考价值的长输入
- 可随时重新读取的低成本证据

## 11.3 会话压缩流程草案

建议未来把压缩流程固定成下面几个步骤，而不是在 `AgentSession` 里临时处理。

1. **判断是否进入压缩窗口**
   - 依据消息条数、估算 token、最近一次压缩时间
   - 第一版可以先不做 token 精算，只做消息数阈值

2. **识别候选区间**
   - 默认只压缩“较早消息”
   - 保留最近若干轮完整消息
   - 排除 protected content 所在区间

3. **生成高保真摘要**
   - 保留用户意图
   - 保留已确认决策
   - 保留关键验证结论

4. **写回 session 状态**
   - 更新 `session_summary`
   - 或更新 `compression_state`
   - 截短 `messages` 到最近窗口

5. **可选地刷新长期记忆**
   - 若发现摘要中有跨会话价值的信息，再提示或自动写入 memory inbox

## 11.4 第一版实现边界建议

为了避免把复杂度一下拉太高，建议第一版只做下面这些：

- 只维护一个 `session_summary`
- 只保留最近 `N` 条消息
- 只在显式触发时压缩，例如未来的 `/context compress` 或内部维护动作
- 不做自动 block 嵌套
- 不做 decompress / recompress
- 不做 token 级精确预算

第一版**明确不做**：

- 类似 DCP 的消息级 block 图
- 复杂的 protected tool call id 跟踪
- 压缩块的可逆恢复
- 自动压缩提醒注入

这样可以先把“会话摘要层”落地，再决定是否继续长成更完整的 compression subsystem。

## 11.5 推荐的实现顺序调整

结合 D8，后续顺序建议进一步细化为：

1. 先完成当前的 context builder + memory 分层
2. 再引入 `session_summary` 的最小压缩版
3. 然后补 protected content 规则
4. 最后才评估是否值得演进到 block 化压缩状态

## 12. 推荐的落地顺序

建议按下面顺序推进，而不是同时做很多项。

### 第一阶段：把上下文层做出来

- 增加本设计文档
- 引入 `PROJECT.md` / `USER.local.md` 约定
- 新建独立上下文装配器
- 明确 `AgentSession` 从装配器取消息，而不是自己直接拼

### 第二阶段：做轻量记忆系统 MVP

- 引入 `MEMORY.md`
- 引入 `memory/` 目录
- 先做“写入规则”和“读取规则”
- 先不做复杂检索和自动摘要

### 第三阶段：做会话摘要与压缩

- 会话持久化增加 `session_summary`
- 只保留最近若干轮消息
- 将更早消息压缩为摘要
- 明确 protected content 规则
- 视复杂度再决定是否演进为 block 化压缩状态

### 第四阶段：评估是否需要 `/prompt`

- 如果发现 skill 和 memory 仍不足以表达临时控制需求，再做 `/prompt`

### 第五阶段：评估是否需要外部检索层

- 当 `memory/`、归档会话、项目文档、会议记录规模明显增长后
- 再评估是否接入 `qmd` 这类本地检索系统
- 若接入，应优先作为按需 retrieval 后端，而不是 system prompt 主装配器

### 第六阶段：环境引导与执行轨迹

受 Meta-Harness (arXiv:2603.28052) 启发，在基础上下文层稳定后增加两项能力：

#### 环境引导（env bootstrap）

- 会话启动前自动采集工作环境快照（目录结构、语言版本、包管理器、系统状态）
- 注入初始提示词，减少 2-4 轮探索性工具调用
- 实现要求：15 秒超时，静默失败，上下文预算不超过 5%

#### 执行轨迹收集（trace collection）

- 结构化记录每次 agent 调用的完整执行轨迹（请求类型、工具序列、token 用量、耗时、结果状态）
- 论文消融实验表明完整执行轨迹对 harness 优化不可替代（中位分 50.0 vs 34.6 仅评分）
- 所有日志使用机器可读格式（JSON），为后续 harness 演进提供数据基础

### 第七阶段：任务类型路由

当基础 harness 稳定且有足够执行轨迹后，考虑按请求类型（bug 修复 / 新功能 / 重构 / 调研）分流不同的工具组合和提示策略。这是更高级的 harness 优化，不急于实现。

## 13. 初步结论

本项目后续应当采用的不是“单一大 prompt 模型”，而是：

> **分层上下文模型**：基础系统层 + 启动背景层 + 长期记忆层 + 场景控制层 + 会话工作层 + 临时证据层。

其中最重要的三个原则是：

1. **长期记忆和当前会话分开。**
2. **skill / session prompt / memory 三者职责分开。**
3. **详细内容按需读取，默认上下文尽量短。**

此外，受 Meta-Harness 研究启发，新增一条原则：

4. **harness 决策（提示构造、工具选择、重试、错误处理）应逐步显式化为可观测、可优化的整体。**

如果后续要讨论实现优先级，建议优先讨论以下三项：

1. 是否采用 `PROJECT.md` + `USER.local.md`。
2. `MEMORY.md` 与 `memory/` 的最小可行结构是什么。
3. 会话压缩是否在当前 JSON 会话存储上继续演进。
