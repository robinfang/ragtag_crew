# `ragtag_crew` 项目 Roadmap

## 1. 项目定位

`ragtag_crew` 当前不是要一次性做成完整的 OpenClaw 全量平替。

更现实、也更适合当前阶段的目标是：

> 先把它做成一个 **单用户、自部署、Telegram 优先、可扩展外部能力的本地 agent 产品**。

这里的“可扩展外部能力”是接下来路线调整后的核心。

它至少包括四类能力：

- 本地工具能力
- `MCP client` 外部工具能力
- `OpenAPI` 工具能力
- 搜索能力（联网搜索 + 平台本地搜索适配）

## 2. 当前状态

当前项目已经具备一套可继续演进的核心底座：

- Telegram 单入口已打通
- 自建 `agent loop` 已可用
- 本地工具系统已具备基础形态
- 超时、会话持久化、流式输出已接入
- `PROJECT.md` / `USER.local.md` / `MEMORY.md` 分层上下文已落地
- 最小 `session_summary` 与 `/context` 已具备
- 最小 `/memory` 闭环已具备

这说明项目已经不再处于“从零起步”的阶段，而是进入：

> **在稳定底座之上，决定下一阶段主线能力的阶段。**

当前新的主线已经很明确：

- 尽快接入 `MCP` 能力
- 强化联网搜索能力
- 后续支持 `OpenAPI` 工具接入
- 在 Windows 下尽量利用 `Everything` 这类本地能力

## 3. 战略判断

接下来项目的重点，不应只是继续深挖上下文系统。

更合理的主线是：

> **先把外部能力层建起来，再让上下文系统承接这些新能力。**

也就是说，接下来不是“继续围绕本地工具打磨”，而是要把项目升级成：

- 一个有本地工具能力的 Telegram agent
- 一个能接外部 `MCP` server 的 agent
- 一个能调用固定 `OpenAPI` 工具的 agent
- 一个既能联网搜索，也能利用宿主机本地搜索能力的 agent

## 4. 总体路线

建议把后续项目路线拆成 8 个阶段。

### 阶段 0：守住现有底座

**目标：**

- 保持当前核心架构稳定，不推翻已有的 `agent loop + tool runtime + context system`

**重点：**

- 不为了接 `MCP` 或 `OpenAPI` 另起一套 agent 架构
- 不把新能力直接散落到 Telegram handler 里
- 保持所有新能力都挂在统一工具系统下

**产出：**

- 后续所有能力都在现有架构上继续生长

### 阶段 1：建立“外部能力接入层”

**目标：**

- 让工具系统从“本地工具集合”升级为“多来源工具系统”

**应形成的工具来源分类：**

- 本地内置工具
- `MCP` 工具
- `OpenAPI` 工具
- 搜索工具
- 平台适配工具

**这一步为什么最重要：**

- 如果没有统一的外部能力接入层，后面 `MCP`、`OpenAPI`、搜索会各走各的逻辑
- 那样会让工具 schema、错误处理、超时、权限边界和结果显示全部分裂

**产出：**

- 一个统一的外部工具接入架构

### 阶段 2：优先补搜索能力

**目标：**

- 让 agent 先具备稳定的“找信息”能力

这一阶段建议拆成两条：

#### 2A. 联网搜索 API

**建议：**

- 优先接稳定搜索 API，而不是先做网页抓取
- 如果后续单独做 search gateway，`ragtag_crew` 第一版应直接接它的 `OpenAPI`

**原因：**

- 搜索 API 在可用性、稳定性、结构化结果和可维护性上都更适合当前项目
- 能更快提供实际价值
- 对当前这套自有生态来说，先多包一层 `MCP adapter` 没有必要

#### 2B. Windows `Everything` 适配

**建议：**

- 将 `Everything` 视为 **Windows 平台本地搜索适配器**
- 不把它混入通用 shell 语义里

**原因：**

- 它本质上是宿主机能力增强，而不是跨平台通用工具
- 更适合独立封装为平台适配模块

**注意：**

- 这里会涉及进程调用，但更推荐直接调用可执行文件，而不是通过 shell 拼字符串

**产出：**

- 稳定的联网搜索能力
- Windows 下显著更强的本地文件搜索能力

### 阶段 3：接入 `MCP client`

**目标：**

- 让 agent 可以连接外部 `MCP` server，并把它们暴露的能力纳入当前工具系统

**当前建议：**

- 先做 `client`
- 不做 `server`

**原因：**

- 这更符合当前产品形态：`ragtag_crew` 自己是 agent，而不是对外提供工具平台

**关键要求：**

- `MCP` 工具要被统一映射到当前工具运行时
- 要有超时、错误、状态展示和启停管理

**产出：**

- `ragtag_crew` 能稳定消费外部 `MCP` server 提供的工具能力

### 阶段 4：接入固定 `OpenAPI` 工具

**目标：**

- 支持少量、明确、预配置的 API 工具

**当前不建议：**

- 不要一开始就做“给一个 spec URL / 文件就动态导入工具”

**原因：**

- 动态导入会立刻引入认证、schema 清洗、参数映射、错误处理和安全边界的大量复杂性
- 当前项目更需要“尽快形成稳定可用能力”，而不是先做平台级泛化

**正确路线：**

- 第一版只支持少量固定 API
- 把认证、请求封装、结果裁剪和错误处理先做顺

**产出：**

- 一套稳定的固定 `OpenAPI` 工具接入机制

### 阶段 5：让上下文系统承接外部结果

**目标：**

- 把 `MCP`、`OpenAPI`、联网搜索、`Everything` 的结果，纳入当前上下文与记忆体系

**必须回答的问题：**

- 哪些结果只属于当前轮临时证据
- 哪些结果应该进入 `session_summary`
- 哪些结果适合被手动或自动写入 `memory`
- 哪些结果需要保留来源信息，方便回溯

**原因：**

- 外部能力接入后，系统复杂度上升最快的地方，不是调用本身，而是“结果如何被后续轮次继续使用”

**产出：**

- 外部结果进入上下文系统的明确规则

### 阶段 6：补控制面与诊断面

**目标：**

- 让用户能看见当前系统接了什么、能做什么、哪里出了问题

**建议覆盖内容：**

- 已连接的 `MCP` server 列表与状态
- 已加载的 `OpenAPI` 工具列表
- 搜索 provider 与平台搜索能力状态
- 最近调用来源、错误与失败原因

**原因：**

- 外部能力一多，没有控制面和诊断面，系统会迅速变得难用且难排查

**产出：**

- 一个可解释、可管理的外部能力控制面

### 阶段 7：回头深化上下文系统

**目标：**

- 在外部能力层稳定之后，再继续增强上下文与记忆系统

**这时再做更合适的内容：**

- `protected content` 规则
- 更强的 `session_summary`
- `/prompt`
- `memory search`
- 再后续评估 `qmd`
- 最后再评估 block 化 compression state

**原因：**

- 这类能力现在仍然重要，但不应先于外部能力层建设

## 5. 优先级排序

建议的整体优先级如下：

1. 外部能力接入层
2. 联网搜索 API
3. Windows `Everything` 适配
4. `MCP client`
5. 固定 `OpenAPI` 工具接入
6. 外部结果进入上下文系统
7. 控制面与诊断面
8. 上下文系统高级增强

这个排序的核心思想是：

> 先补“能力来源”，再补“能力承接”，最后补“能力治理”。

## 6. 里程碑建议

### M1：统一外部工具底座

- 项目能承载多来源工具，而不是只有本地工具

### M2：稳定联网搜索

- agent 可以稳定做互联网搜索，不依赖网页抓取 hack

### M3：Windows 本地搜索增强

- 在 Windows 环境下能利用 `Everything` 做高效本地检索

### M4：可用的 `MCP client`

- 可以连接一个或多个外部 `MCP` server 并稳定调用工具

### M5：可用的固定 `OpenAPI` 工具

- 可以稳定调用少量预配置 API

### M6：外部结果进入上下文系统

- 搜索结果、`MCP` 结果、`OpenAPI` 结果都能被当前会话继续利用

### M7：控制与诊断能力可用

- 用户能看见系统状态、能力来源和失败原因

### M8：上下文系统高级能力成熟

- 再继续做更精细的记忆、压缩与上下文治理

## 7. 当前明确不做的事项

为了避免项目范围失控，当前建议明确暂不做：

- `MCP server`
- 动态 `OpenAPI spec` 导入平台
- `qmd` 直接接入主链路
- block 化 compression state
- 图片 / 文件输入
- 多前端抽象

这些事项并不是永久不做，而是：

> **不应早于当前的外部能力层建设。**

## 8. 几个关键决策的当前结论

### 8.1 `MCP`

- 先做 `client`
- 暂不做 `server`

### 8.2 联网搜索

- 先走稳定搜索 API
- 不先做网页抓取

### 8.3 `OpenAPI`

- 第一版做少量固定 API
- 不做动态 spec 导入

### 8.4 Windows `Everything`

- 视为平台专用搜索适配器
- 不归入通用 shell 本身

### 8.5 上下文系统

- 继续保留当前分层上下文路线
- 但项目主线优先级下调到“承接外部能力结果”这一层

## 9. 一句话结论

`ragtag_crew` 接下来的主线，不应再被理解成“继续补一个本地 Telegram coding agent”。

更准确的方向是：

> **先把它做成一个能接外部能力的 Telegram agent，再让上下文系统去稳定承接这些新能力。**

这才是当前阶段最符合产品价值、也最容易形成差异化的路线。
